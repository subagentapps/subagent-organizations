## TL;DR

There is **no contributor named "releasebot"** in `anthropic-cli`. The bot you saw is
**[`stainless-app[bot]`](https://github.com/apps/stainless-app)** — the GitHub App owned by
[Stainless](https://www.stainless.com), the same company that generates Anthropic's official
SDKs from OpenAPI specs. It does ~all of the actual release work in this repo.

The actual *release publishing* is handled by **GoReleaser** plus a separate, custom GitHub App
referenced only as `RELEASE_APP_ID` / `RELEASE_APP_PRIVATE_KEY` in repo secrets — that one signs
binaries with an Apple Developer cert, notarizes them with Apple's notary service, and pushes
formula updates to `anthropics/homebrew-tap`.

## What `stainless-app[bot]` actually does

Three observable behaviors in this repo:

1. **Codegen commits.** Pushes commits like:
   - `chore(internal): codegen related update`
   - `codegen metadata`
   - `chore(client): config fixes`
   - `feat(api): CMA Memory public beta`
   - `fix(api): restore missing features`

   These are mechanical regenerations of the Go client when the upstream OpenAPI spec changes.
   Author + committer are both the bot.

2. **Release commits.** Pushes commits literally titled `release: 1.3.2`, `release: 1.3.1`, etc.
   These bump version files and update `CHANGELOG.md`. The bot composes these from the
   conventional-commit history since the previous release (same model release-please uses).

3. **Triggers other workflows.** Listed under "Committed by `stainless-app[bot]`" alongside
   PRs authored by humans — meaning the bot is the actor merging PRs to `main` after CI passes.

## The Stainless GitHub App + the `trigger-release-please` action

The Stainless team ships a public Action that wraps their internal release-please-equivalent:

```yaml
# .github/workflows/create-releases.yml
name: Create releases
on:
  push:
    branches:
      - main

jobs:
  release:
    if: github.ref == 'refs/heads/main' && github.repository == 'anthropics/anthropic-cli'
    runs-on: ubuntu-latest
    permissions: {}
    steps:
      - uses: stainless-api/trigger-release-please@656ec30d61e731fcd70e83ac2084a4040770a504
        id: release
        with:
          repo: ${{ github.event.repository.full_name }}
          stainless-api-key: ${{ secrets.STAINLESS_API_KEY }}
```

Note the `permissions: {}` — the workflow itself has zero GitHub permissions. The
`stainless-app[bot]` GitHub App is what holds the write permissions. Workflow just signals
"I want a release on this repo," and the Stainless backend (authenticated via `STAINLESS_API_KEY`)
takes over from there: opens the release PR, runs the version bump, all under the bot identity.

This is the **same pattern Anthropic uses across all SDK repos** (`anthropic-sdk-typescript`,
`anthropic-sdk-python`, etc.). One Stainless backend, one bot identity, many repos.

## The publish step — a *second* GitHub App + GoReleaser

After the release PR merges and a `v*` tag is pushed, `publish-release.yml` runs:

```yaml
# .github/workflows/publish-release.yml — abridged
name: Publish Release
on:
  push:
    tags: ["v*"]

jobs:
  verify:                                     # 1. ancestry check, no secrets touched
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - name: Verify release commit is on main's first-parent history
        run: |
          git fetch origin main
          if ! git rev-list --first-parent origin/main | grep -qxF "$GITHUB_SHA"; then
            echo "::error::Refusing to release: tag is not on main's first-parent history."
            exit 1
          fi

  goreleaser:
    needs: verify
    runs-on: ubuntu-latest
    environment: production-release          # 2. GitHub Environment gates the secrets

    steps:
      - name: Generate release app token (this repo)
        uses: actions/create-github-app-token@v3
        with:
          app-id: ${{ secrets.RELEASE_APP_ID }}
          private-key: ${{ secrets.RELEASE_APP_PRIVATE_KEY }}

      - name: Generate release app token (homebrew-tap)
        uses: actions/create-github-app-token@v3
        with:
          app-id: ${{ secrets.RELEASE_APP_ID }}
          private-key: ${{ secrets.RELEASE_APP_PRIVATE_KEY }}
          owner: anthropics
          repositories: homebrew-tap

      - uses: goreleaser/goreleaser-action@v7   # 3. GoReleaser does the actual publish
        with:
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ steps.app-token.outputs.token }}
          HOMEBREW_TAP_GITHUB_TOKEN: ${{ steps.tap-token.outputs.token }}
          MACOS_SIGN_P12: ${{ secrets.MACOS_SIGN_P12 }}
          MACOS_SIGN_PASSWORD: ${{ secrets.MACOS_SIGN_PASSWORD }}
          MACOS_NOTARY_ISSUER_ID: ${{ secrets.MACOS_NOTARY_ISSUER_ID }}
          MACOS_NOTARY_KEY_ID: ${{ secrets.MACOS_NOTARY_KEY_ID }}
          MACOS_NOTARY_KEY: ${{ env.MACOS_NOTARY_KEY }}      # base64-encoded .p8
```

Key safety properties worth copying:

1. **Two-job split with `needs:`** — the unprivileged `verify` job runs first with no secrets and
   no environment. Only if it passes does `goreleaser` get the production environment + tokens.
   A bad tag never even prompts the deployment-approval gate.
2. **First-parent ancestry check** — `git rev-list --first-parent` rejects any tag that points
   at a commit *not on the linear main history*. Squashed-PR-only repos already get this
   property; this defends against the merge-commit setting flipping later.
3. **Custom GitHub App for publishing**, distinct from `stainless-app[bot]`:
   - `RELEASE_APP_ID` / `RELEASE_APP_PRIVATE_KEY` are organization-owned secrets.
   - One app, two scopes: this repo (for tag → release) and `anthropics/homebrew-tap`
     (for formula bumps). The `actions/create-github-app-token@v3` action mints separate
     short-lived tokens per scope.
4. **Tag creation is restricted via a repository tag ruleset** to the Stainless App identity —
   commented in the workflow as: *"Tag creation is restricted to the Stainless App via a
   repository tag ruleset."* Means a human can't accidentally cut a release by pushing a tag.
5. **macOS signing + notarization done in-pipeline.** GoReleaser drives `codesign` + Apple's
   notary service. The `.p8` private key is base64-re-encoded immediately before passing it to
   `goreleaser` because GoReleaser's notarize step expects base64; the encoded form is masked
   in logs with `::add-mask::`.
6. **Pinned action SHAs everywhere** — every `uses:` is a 40-char SHA, not a tag. Defends
   against an action publisher pushing a malicious tag.

## What we should copy in `subagent-organizations`

We don't need GoReleaser (no Go binaries) or macOS signing (no app to ship), but several patterns
fit our scale:

| Pattern | Worth copying? | Why |
|---|---|---|
| Two-job verify-then-publish split | ✅ Yes — even for npm | Cheap insurance against bad tags |
| First-parent ancestry check on release tags | ✅ Yes | Three lines of bash |
| Pinned action SHAs (no `@v1` tags) | ✅ Yes | Already a Dependabot-friendly convention |
| `permissions: {}` minimal default + scoped per-job | ✅ Yes | Zero-trust workflow security |
| Custom GitHub App for releases instead of GITHUB_TOKEN | ❓ Eventually | Worth it if/when we cross orgs (e.g., publishing to a homebrew tap repo) |
| Stainless to generate code | ❌ No | We're not driving from an OpenAPI spec |
| GitHub Environment gating production secrets | ✅ Yes | Free with `environment: production-release` |
| Restricting tag creation via ruleset | ✅ Yes when we have a release bot | Prevents accidental releases |

## Putting it next to our current setup

Our repo currently uses Google's `release-please` action (the original; Stainless's is a
fork/adaptation). Same pattern, different owner: bot opens release PR → human merges → tag is
cut → publish workflow runs.

The two main upgrades inspired by anthropic-cli's pipeline that I'd suggest later:
1. Add a **`verify` job** in front of any future publish workflow (blocks tags off main's first-parent line).
2. Pin all GitHub Action references by **SHA, not tag**.

## What appears in the `homebrew-tap` repo (the *third* bot)

Earlier sections covered `stainless-app[bot]` (codegen + release PRs on the source repo)
and the custom `RELEASE_APP_*` GitHub App (token brokering for the publish job). But the
identity that **actually shows up as the commit author on `anthropics/homebrew-tap`** is a
*third* bot: **[`goreleaserbot`](https://github.com/goreleaserbot)** (GitHub user ID 29843943,
`bot@goreleaser.com`).

This matters because the custom `RELEASE_APP_*` App is just the **token broker** — it mints a
short-lived token scoped to the tap repo, but `goreleaser` then uses that token to push a commit
whose **author** it sets to `goreleaserbot`. The App identity never appears in the tap's `git log`.

### Three distinct bots, one pipeline

| Bot | Identity type | Where it commits | What it does |
|---|---|---|---|
| `stainless-app[bot]` | GitHub App (ID 378072) | `anthropics/anthropic-cli` | Codegen, version bumps, opens release PRs, CHANGELOG |
| Custom `RELEASE_APP_*` (no public name) | Org-owned GitHub App | Mints tokens, never directly authors commits | Token brokering for tag→publish *and* tap pushes |
| `goreleaserbot` | Plain GitHub user | `anthropics/homebrew-tap` | Cask formula updates |

### The 4-commit history of `homebrew-tap` shows a security glow-up

The repo is tiny — just 4 commits, no `.github/workflows/` of its own — but the pattern shifted
mid-stream:

```
67bd3fd  PR #3  AlePouroullis (human)  Co-authored-by: goreleaserbot   "Brew cask update for ant version v1.3.2"
03b7c5d  PR #2  AlePouroullis (human)                                  "Add CODEOWNERS requiring @anthropics/sdk approval"
c7d7f67  PR #1  packyg       (human)   Co-authored-by: goreleaserbot   "Brew cask update for ant version v1.2.1"
0e6080c  direct goreleaserbot                                          "Brew cask update for ant version v1.0.0"
```

1. **v1.0.0**: `goreleaserbot` pushed straight to `main`. No PR, no review.
2. **PR #2 (Apr 23)**: a human added `CODEOWNERS` requiring `@anthropics/sdk` review. The PR
   description is the gold:
   > *"The L3 ruleset on this repo has require_code_owner_review enabled, but without a
   > CODEOWNERS file that requirement has nothing to enforce and any single writer can approve."*
3. **v1.2.1, v1.3.2**: now the bot **opens a PR** instead of pushing direct, and the human
   merging it picks up `goreleaserbot` as `Co-authored-by:`.

The control point that broke and was patched: a GitHub branch ruleset with
`require_code_owner_review = true` does **nothing** unless a `CODEOWNERS` file exists. The bot's
PAT/App had write access; without an effective code-owner gate it could land changes unilaterally.
Adding `CODEOWNERS` made the rule actually bite.

### Where the v1.0.0 → v1.2.1 transition happens in the publish workflow

In [`anthropic-cli`'s `publish-release.yml`](https://github.com/anthropics/anthropic-cli/blob/main/.github/workflows/publish-release.yml),
the goreleaser step has two scoped tokens:

```yaml
- name: Generate release app token (homebrew-tap)
  uses: actions/create-github-app-token@v3
  with:
    app-id: ${{ secrets.RELEASE_APP_ID }}
    private-key: ${{ secrets.RELEASE_APP_PRIVATE_KEY }}
    owner: anthropics
    repositories: homebrew-tap                             # ← scoped to JUST the tap

- uses: goreleaser/goreleaser-action@v7
  env:
    HOMEBREW_TAP_GITHUB_TOKEN: ${{ steps.tap-token.outputs.token }}
```

`goreleaser`'s `brews:` config (in `.goreleaser.yml`, not shown but implied) uses
`HOMEBREW_TAP_GITHUB_TOKEN` to push the formula update. Whether that push is a **direct commit**
or a **PR** is controlled by goreleaser's `pull_request:` block — the v1.0.0 → v1.2.1 jump
corresponds to flipping `pull_request: enable: true` in that config (see
`f1f4c5a fix(goreleaser): correct pull request config` in the source repo's history).

## Glossary

- **`stainless-app[bot]`** — Stainless's organization-owned GitHub App. Public profile,
  installable by anyone using their service.
- **`goreleaserbot`** — A plain GitHub user account (not an App) that GoReleaser uses as the
  default `git config user.name`/`email` for any commits it pushes. Authentication happens via
  whatever token GoReleaser is given (usually `HOMEBREW_TAP_GITHUB_TOKEN`); the *identity* on the
  resulting commit is always `goreleaserbot <bot@goreleaser.com>`.
- **`web-flow`** — GitHub's own bot that signs merge commits made via the web UI. Co-committer
  on most human-authored PRs.
- **`packyg`, `meorphis`, `RobertCraigie`, `AlePouroullis`, `bruce-hill`** — humans at
  Anthropic + Stainless contributing CI/pipeline work.
